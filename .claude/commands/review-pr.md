# Review Pull Request

GitHub PR을 체계적으로 리뷰하고 코멘트를 작성해주세요.

## 작업 순서

1. **PR 정보 조회**

    - `gh pr view {pr_number} --json title,body,headRefName,baseRefName,state,files,additions,deletions` 명령어로 PR 기본 정보 확인
    - `gh pr diff {pr_number}` 명령어로 변경사항 확인

2. **원래 브랜치 저장**

    - 현재 브랜치를 기억해두기 (나중에 복귀용)

3. **PR 브랜치로 체크아웃**

    - `gh pr checkout {pr_number}` 명령어로 PR 브랜치로 전환

4. **변경된 파일 분석**

    - 주요 변경 파일들을 Read 도구로 읽기
    - 코드 품질, 아키텍처, 잠재적 버그 검토
    - 타입 체크 확인 (필요시 `npm run type-check`)

5. **리뷰 코멘트 작성**

    리뷰는 다음 구조로 작성:

    ```markdown
    ## 리뷰 완료 ✅

    ### 👍 좋은 점

    - 긍정적인 부분 간단히

    ### ⚠️ 주요 우려사항

    - 핵심 이슈만 간결하게
    - 필요시 코드 예시 (최소화)

    ### 📝 필수 조치 (머지 전)

    1. 반드시 해결해야 할 사항

    ### 💡 권장 사항 (머지 후 별도 PR)

    1. 개선하면 좋을 사항

    ### 🎯 결론

    - 승인/조건부 승인/반려
    ```

    **중요:**

    - 코드 디테일은 최소화 (작성자가 알만한 내용은 생략)
    - 읽기 쉽게 작성
    - 한국어 사용

6. **PR에 코멘트 작성**

    - `gh pr comment {pr_number} --body "리뷰 내용"`
    - 마크다운 형식으로 작성

7. **원래 브랜치로 복귀**
    - 저장해둔 원래 브랜치로 체크아웃

## 리뷰 체크리스트

### 코드 품질

- [ ] 타입 에러 없음
- [ ] 테스트 포함 여부
- [ ] 성능 이슈 없음

### 아키텍처

- [ ] 파일/컴포넌트 크기 적정
- [ ] 관심사 분리 적절
- [ ] 재사용성 고려
- [ ] 의존성 관리 적절

### 잠재적 문제

- [ ] 메모리 누수 가능성
- [ ] Race condition
- [ ] 에러 핸들링
- [ ] 엣지 케이스 처리

### 프로젝트 규칙

- [ ] CLAUDE.md 및 .cursor/rules 준수
- [ ] 다국어 처리 (UI 텍스트)
- [ ] 로깅 사용 (/debug 라이브러리)
- [ ] 문서화 필요 여부

## 주의사항

- **절대 Prettier 포맷팅 체크는 하지 않기** (사용자가 직접 실행)
- 코드 디테일 최소화 (핵심만 언급)
- 작성자가 이미 알고 있을 내용은 생략
- 건설적이고 친절한 톤 유지
- 구체적인 개선 방향 제시

## 사용 예시

```
/review-pr 1199
```

PR 번호를 인자로 받습니다.
